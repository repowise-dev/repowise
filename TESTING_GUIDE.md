# repowise — Local Testing Guide

Test repowise end-to-end against a real repo (`interview-coach`) using:
- **LLM**: `gemini-3-flash-preview` via LiteLLM
- **Embeddings**: `gemini-embedding-001` via GeminiEmbedder
- **Frontend**: Next.js 15 at `http://localhost:3000`
- **API server**: FastAPI at `http://localhost:7337`

## Step 3 — Index the `interview-coach` Repo

This crawls all source files, parses symbols, builds a call graph, and generates wiki pages using Gemini:

```powershell
cd C:\Users\ragha\Desktop\repowise

uv run repowise init "C:\Users\ragha\Desktop\interview-coach" `
    --provider litellm `
    --model "gemini/gemini-3-flash-preview" `
    --yes
```

**What happens during init:**
1. Traverses all source files (Python, JS/TS, etc.)
2. Parses symbols (functions, classes, methods)
3. Builds dependency/call graph
4. Indexes git history per file
5. Detects dead code
6. Estimates token costs
7. **Calls Gemini** to generate wiki pages for each file/module
8. Persists everything to `.repowise/wiki.db` + `.repowise/index.fts`

Expected output:
```
Scanning interview-coach ...  [████████████] 100%
Parsed 142 symbols across 38 files
Generating wiki pages ...     [████████████] 100%
Done. 38 pages written to .repowise/wiki.db
```

> **Cost note**: Init calls Gemini for each file. A ~40-file Python repo costs roughly 50K–200K tokens with `gemini-3-flash-preview` (well within free tier limits).

## Step 6 — End-to-End Test Walkthrough

### 6.1 Dashboard

Navigate to `http://localhost:3000`. You should see:
- **Total Pages** — count of generated wiki pages
- **Fresh / Stale / Outdated** — freshness breakdown
- **Repos list** — `interview-coach` repo card
- **Recent Jobs** — the init job with status `completed`

### 6.2 Repo Overview

Click the `interview-coach` repo card. You should see:
- Git metadata: head commit, branch, last commit date
- Overview wiki page (generated by Gemini)
- File and symbol counts

### 6.3 Wiki Pages

Click any page in the sidebar. Each page shows:
- **Confidence badge** (fresh/stale/outdated based on git recency)
- **Content** — Gemini-generated Markdown documentation
- **Git history** panel — recent commits for this file
- **Symbols panel** — functions/classes defined in the file
- **Metadata** — token count, model used, generation timestamp

### 6.4 Search

Go to `http://localhost:3000/search` and try:
- `"auth"` — finds pages related to authentication
- `"database"` — finds DB-related modules
- `"websocket"` — finds real-time code

The search bar hits `/api/search` with `search_type=hybrid` (FTS + semantic).

### 6.5 Symbol Lookup

Go to `http://localhost:3000/symbols?repo_id=<repo-id>`. Use the filter to:
- Filter by language (Python, TypeScript, etc.)
- Filter by kind (function, class, method)
- Search by name

### 6.6 Dependency Graph

Go to `http://localhost:3000/graph?repo_id=<repo-id>`. The D3 force-directed graph shows:
- Nodes = files/modules
- Edges = import/call relationships
- Click a node to see its wiki page

## Step 7 — API Smoke Tests

Run these directly in PowerShell to verify backend correctness:

```powershell
$base = "http://localhost:7337"

# List all repos
Invoke-RestMethod "$base/api/repos"

# Set repo_id from the response above, e.g.:
$repoId = (Invoke-RestMethod "$base/api/repos")[0].id

# Get repo details
Invoke-RestMethod "$base/api/repos/$repoId"

# List wiki pages
Invoke-RestMethod "$base/api/pages?repo_id=$repoId&limit=5"

# Full-text search
Invoke-RestMethod -Method Post "$base/api/search" `
    -ContentType "application/json" `
    -Body '{"repo_id": "'"$repoId"'", "q": "auth", "search_type": "fts", "limit": 5}'

# Semantic search (uses GeminiEmbedder)
Invoke-RestMethod -Method Post "$base/api/search" `
    -ContentType "application/json" `
    -Body '{"repo_id": "'"$repoId"'", "q": "authentication flow", "search_type": "semantic", "limit": 5}'

# List symbols
Invoke-RestMethod "$base/api/symbols?repo_id=$repoId&limit=10"

# Get dependency graph
Invoke-RestMethod "$base/api/graph/$repoId"

# Git hotspots
Invoke-RestMethod "$base/api/git/$repoId/hotspots"

# Dead code
Invoke-RestMethod "$base/api/dead-code?repo_id=$repoId"
```

## Step 8 — Test Semantic Search Specifically

Semantic search uses GeminiEmbedder (`gemini-embedding-001`) at query time. To verify:

```powershell
Invoke-RestMethod -Method Post "http://localhost:7337/api/search" `
    -ContentType "application/json" `
    -Body (@{
        repo_id     = $repoId
        q           = "how does the interview scoring system work"
        search_type = "semantic"
        limit       = 5
    } | ConvertTo-Json)
```

If you see ranked results with `score` fields > 0.7, semantic search is working correctly.

If you see an error like `GEMINI_API_KEY not set`, double-check Step 1.

## Troubleshooting

### `ModuleNotFoundError: No module named 'google'`
```powershell
uv pip install google-genai
```

### `ModuleNotFoundError: No module named 'litellm'`
```powershell
uv pip install litellm
```

### Server shows `MockEmbedder` in logs / semantic search returns empty
Check that `REPOWISE_EMBEDDER=gemini` is set in the PowerShell session running `repowise serve`.

### `repowise: command not found` in PowerShell
Always prefix with `uv run`:
```powershell
uv run repowise --help
```

### Init fails with `rate limit` errors
Add `--concurrency 1` to slow down Gemini calls:
```powershell
uv run repowise init "C:\Users\ragha\Desktop\interview-coach" `
    --provider litellm `
    --model "gemini/gemini-3-flash-preview" `
    --concurrency 1 `
    --yes
```

### Frontend shows "Failed to fetch repos"
Ensure:
1. Server is running on port 7337
2. `REPOWISE_API_URL=http://localhost:7337` is set before starting `npm run dev`
3. CORS is allowed (it is — server allows all origins by default)

### `REPOWISE_DB_URL` path issues on Windows
Use forward slashes in the path:
```powershell
$env:REPOWISE_DB_URL = "sqlite+aiosqlite:///C:/Users/ragha/Desktop/interview-coach/.repowise/wiki.db"
```

## Quick Reference — All Env Vars

| Variable | Value | Purpose |
|----------|-------|---------|
| `GEMINI_API_KEY` | `AIzaSy...` | Authenticates both LiteLLM and GeminiEmbedder |
| `REPOWISE_PROVIDER` | `litellm` | Selects LiteLLM as the LLM provider for init/sync |
| `REPOWISE_MODEL` | `gemini/gemini-3-flash-preview` | Model name in LiteLLM format |
| `REPOWISE_EMBEDDER` | `gemini` | Activates GeminiEmbedder in the server |
| `REPOWISE_DB_URL` | `sqlite+aiosqlite:///...` | Points server at the repo's wiki.db |
| `REPOWISE_API_URL` | `http://localhost:7337` | Frontend → backend URL |
| `REPOWISE_EMBEDDING_DIMS` | `768` | Optional: override embedding dimensions (default 768) |
